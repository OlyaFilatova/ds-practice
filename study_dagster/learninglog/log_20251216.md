- [x] https://docs.dagster.io/getting-started/concepts
    - Asset
        - a logical unit of data such as a table, dataset, or machine learning model
        - can have dependencies on other assets
        - can interact with many other Dagster entities
        - the definition is automatically added to a top-level Definitions object
        - deep dive - https://dagster.io/blog/software-defined-assets
        - relationships
            - is described by an asset spec
            - may be programmatically built by a component
            - is added to a top-level Definitions object to be deployed
            - may use an asset check, a config, an io manager, a partition, a resource
            - may be used in a job, a schedule, a sensor
    - Asset check
        - ensures an asset meets certain expectations around data quality, freshness or completeness
        - relationships
            - may be used by an asset
            - is added to a top-level Definitions object to be deployed
    - Asset spec
        - standalone objects that describe the identity and metadata of Dagster entities without defining their behavior
        - relationships
            - may describe the identity and metadata of an asset
    - Code location
        - a collection of Dagster entity definitions deployed in a specific environment
        - determines the Python environment
        - A Dagster project can have multiple code locations
        - relationships
            - must contain at least one top-level Definitions object
    - Component
        - objects that programmatically build Dagster entity definitions
        - accept schematized configuration parameters
        - designed to help quickly bootstrap parts of a Dagster project
        - serve as templates for repeatable patterns
        - relationships
            - builds assets, asset_checks, jobs, schedules, sensors, resources and other definitions
    - Config
        - used to specify config schema for assets, jobs, schedules, and sensors
        - A RunConfig is a container for all the configuration that can be passed to a run
        - relationships
            - may be used by an asset, a job, a schedule, a sensor
    - Definitions
        1. objects that combine metadata about Dagster entities with Python functions that define how they behave
        2. top-level Definitions object that contains references to all the definitions in a Dagster project
        - relationships
            - Top-level Definitions object 
                - may contain one or more asset, asset check, io manager, job, resource, schedule, sensor definitions
            - definition
                - may be the output of a component
                - must be deployed in a code location
    - Graph
        - connects multiple ops together to form a DAG
        - when using assets, you will not need to use graphs directly
        - relationships
            - graph may use a config
            - graph must include one or more ops
            - graph must be part of job to execute
    - IO manager
        - defines how data is stored and retrieved between the execution of assets and ops
        - relationships
            - may be used by an asset
            - is added to a top-level Definitions object to be deployed
    - Job
        - a subset of assets or the GraphDefinition of ops.
        - are the main form of execution in Dagster.
        - relationships
            - may contain a selection of assets
            - may use a config
            - may contain a graph
            - may be used by a schedule, a sensor
            - is added to a top-level Definitions object to be deployed
    - Op
        - a computational unit of work.
        - are arranged into a GraphDefinition to dictate their order.
        - have largely been replaced by assets.
        - relationships
            - may use a type
            - must be contained in graph to execute
    - Partition
        - represents a logical slice of a dataset or computation mapped to a certain segments (such as increments of time)
        - enable incremental processing
        - relationships
            - may be used by an asset
    - Resource
        - a way to make external resources (like database or API connections) available to Dagster entities during job execution
        - A ConfigurableResource is a resource that uses structured configuration.
        - relationships
            - may be used by an asset, a schedule, a sensor
            - is added to a top-level Definitions object to be deployed
    - Type
        - a way to define and validate the data passed between ops
        - relationships
            - may be used by an op
    - Schedule
        - a way to automate jobs or assets to occur on a specified interval
        - the schedule can be set with a run configuration (RunConfig) to pass parameters
        - relationships
            - may include a job or selection of assets
            - may include a config if the job or assets include a config
            - is added to a top-level Definitions object to be deployed
    - Sensor
        - a way to trigger jobs or assets when an event occurs
        - the sensor can be set with a run configuration (RunConfig) to pass parameters
        - relationships
            - may include a job or selection of assets
            - may include a config if the job or assets include a config
            - is added to a top-level Definitions object to be deployed

    - `create-dagster project project-name` - to scaffold a package containing a Dagster Definitions object
    - `dg dev` to launch project locally at http://127.0.0.1:3000
- [x] https://docs.dagster.io/dagster-basics-tutorial/assets
    - primary building blocks
    - represent the underlying entities in our pipelines, such as database tables, machine learning models, or AI processes
    - example scaffolding command `dg scaffold defs dagster.asset assets.py`
    - next define an asset using the @dg.asset
    - `dg check defs` - to check that the Definitions object can be loaded without error as new Dagster objects are added
    - materialize assets
        - using ui
        - using cli
            - all assets `dg launch --assets "*"`
            - specific assets `dg launch --assets customers,orders,payments`
- [x] https://docs.dagster.io/dagster-basics-tutorial/resources
    - Resources are Dagster objects much like assets, but they are not executed
    - resources are reusable objects that supply external context, such as database connections, API clients, or configuration settings
    - example scaffolding command `dg scaffold defs dagster.resources resources.py`
    - example defines the DuckDB resource
    - a resources function is decorated with the @dg.definitions
    - updating assets to use the resource
        - set the resource as a parameter in each asset, its name matches the key that was set when defining the resource. 
- [x] https://docs.dagster.io/dagster-basics-tutorial/dependencies
    - To link the assets together, set the `deps` parameter within the asset decorator.
- [x] https://docs.dagster.io/dagster-basics-tutorial/asset-checks
    - define a function and decorate it with the @dg.asset_check
        - Within the asset check, set the asset parameter to indicate which asset the check runs against.
    - use dg.AssetCheckResult to set if check has passed and additional metadata
- [x] https://docs.dagster.io/dagster-basics-tutorial/schedules
    - There are several ways to automate assets in Dagster: scheduled and event-driven pipelines
    - schedules exist within the Definitions layer
    - example scaffolding command `dg scaffold defs dagster.schedule schedules.py`
    - Schedules consist of a cron_schedule and target
- [x] https://docs.dagster.io/dagster-basics-tutorial/custom-components
    - Components generate Definitions through a configuration layer (helps avoid duplications)
    - There are built-in components
    - example scaffolding command `dg scaffold component Tutorial`
    - The first step is to create a dg.Model for the ETL assets.
        - dg.Model turns any class that inherits from it into a Pydantic model
        - This model will contain the attributes that define an asset
    - Next, add the interface to the dg.Component class
    - In build_defs method construct a Definitions object containing the Dagster objects
    - Now following command can be used to scaffold definitions `dg scaffold defs dagster_tutorial.components.tutorial.Tutorial tutorial`
        - update the YAML file created when you scaffolded a definition from the component
            - type - component class
            - attributes - contains parameters described in the component interface
